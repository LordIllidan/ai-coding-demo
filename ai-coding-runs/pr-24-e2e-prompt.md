Fixed CI trx generation bug in test project config (couldn't touch `.github/workflows/dotnet-ci.yml` — guard-agent-scope job hard-blocks AI branches from modifying it).

**Root cause**: `dotnet test PolicyPlatform.slnx --logger "trx;LogFileName=test-results.trx"` runs all 3 test projects (Domain.Tests, Application.Tests, Api.E2E.Tests) in one solution-level invocation. With no `--results-directory` set, VSTest resolves the trx output relative to the shared working directory the solution-level command runs from, not per-project — so all three projects contend for the same `test-results.trx` path and can clobber/drop each other's report. That leaves the "Publish test results" step's `**/test-results.trx` glob matching nothing → `No test report files were found`.

**Fix**: `tests/Directory.Build.props` (new file) — sets `VSTestResultsDirectory` to a path namespaced by `$(MSBuildProjectName)`, so each test project writes its `test-results.trx` to its own distinct folder (`tests/TestResults/<ProjectName>/test-results.trx`). All three still match the existing `**/test-results.trx` glob in the unmodified workflow, but no longer collide.

Scope note: I read through every file touched by AISDLC-40/41 (domain/application/API/tests for the theft-claim + police-report-number feature) and found no compile-breaking mismatches, so build failure wasn't the cause. I couldn't execute `dotnet` locally (blocked by this agent's sandbox permissions — build/test execution is reserved for the unittest/e2e workers), so this fix is based on static analysis of the known VSTest multi-project trx collision behavior, not a live repro. Flagging that for the review/test workers to confirm on the next CI run.
